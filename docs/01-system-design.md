# システム設計書 — デルカク✓

## 1. 概要

| 項目 | 内容 |
|---|---|
| アプリ名 | デルカク✓（出るまえ確認） |
| サブタイトル | 出発前の忘れ物チェック |
| 目的 | 出発前の最終確認を簡単・確実に行い、忘れ物を防止する |
| 形態 | PWA（単体アプリ、共通エンジンは使用しない） |
| バックエンド | なし（サーバーレス、全データはローカル保存） |
| 主要ターゲット | 忘れ物が多い人／学生／社会人／子育て世代／高齢者／ADHD傾向のある人／通院・介護で持ち物管理が必要な人 |

非機能要件として「シンプル・迷わない・大きな文字・大きなボタン・高齢者でも使える・3タップ以内で主要操作完了」を最優先する。

│        │                                 │
│        ▼                                 │
│  ┌───────────────────────────────────┐  │
│  │   状態管理層 (Zustand)              │  │

サーバーサイドの処理は存在せず、すべての機能はクライアント（端末内）で完結する。GitHub Pagesは静的ファイルの配信のみを担当する。

## 3. 技術スタック

| レイヤー | 技術 | 採用理由 |
|---|---|---|
| UI | React 18 | コンポーネント単位での開発、将来のCapacitor移行との親和性 |
| ビルド | Vite | 高速な開発体験、PWAプラグインとの統合が容易 |
| 言語 | TypeScript | データ構造（チェック項目・セット・予定）の型安全性を確保し、不整合バグを防ぐ |
| PWA化 | `vite-plugin-pwa`（Workbox） | Service Worker・マニフェスト生成を自動化 |
| 状態管理 | Zustand | Reduxより記述量が少なく、localStorage永続化ミドルウェアが標準サポートされている |
| 永続化 | localStorage（`zustand/middleware` の `persist`） | 初期リリースの要件に合致。将来IndexedDBへ移行しやすい抽象化を設ける |
| ルーティング | React Router | 画面数が10前後あるためSPA内ルーティングが必要 |
| 通知 | Web Notifications API + Service Worker | サーバー不要で実装可能な範囲の通知（詳細は5章） |
| スタイリング | CSS Modules または Tailwind CSS | 「大きな文字・大きなボタン」を統一管理するデザイントークンが必要 |
| デプロイ | GitHub Pages（GitHub Actions による自動デプロイ） | 既存プロジェクト群と同じ運用方法 |

## 5. 通知機能の技術方針（重要な制約事項）

ブラウザ標準のPWAには、サーバーなしで「アプリを閉じていても指定時刻に確実に通知する」ことに技術的な制約がある。本プロジェクトはサーバーを持たない方針のため、以下の段階的な対応を取る。

### 5.1 MVPでの実装方針
- `Notification` APIで通知許可をリクエストし、Service Worker経由で通知を表示する。
- アプリ起動中・バックグラウンドタブにある間は `setTimeout` / `Periodic Background Sync`（対応ブラウザのみ）でスケジュール通知を行う。
- iOS Safari は「ホーム画面に追加（PWAインストール）」かつ iOS 16.4以降でのみ通知に対応するため、オンボーディングで明示的に案内する。
- アプリが完全に閉じられている場合、ブラウザ・OSの制約により通知が届かないケースがある（特にiOS、Android端末のバッテリー最適化機能）。この制約はユーザーに正直に伝え、「通知が来なかった場合に備えてアプリを開いた際に未完了チェックを表示する」フォールバックUIで補完する。
- 詳細な代替案・将来対応は [07-improvements.md](07-improvements.md) を参照。

### 5.2 将来構想との関係
Capacitorによるネイティブアプリ化（将来構想）を行えば、OS標準のローカル通知APIを使用でき、上記の制約は解消される。データ構造（4章参照）は、この移行を見据えて「通知スケジュール」をアプリ層の汎用モデルとして設計し、通知の発火方法（Web Notification / ネイティブ）だけを差し替えられるようにする。

## 6. データ永続化方針

- 初期リリース: `localStorage` に、ストアごとにJSONを保存（キー設計は[04-data-structure.md](04-data-structure.md)参照）。
- スキーマには `schemaVersion` を持たせ、将来の構造変更時にマイグレーション関数で吸収する。
- 永続化処理は `src/lib/storage.ts` に抽象化し、将来 `IndexedDB` や `Capacitor Preferences/SQLite` に切り替える際もストア層のコードは変更不要にする。

## 7. 非機能要件

| 区分 | 要件 |
|---|---|
| アクセシビリティ | 文字サイズ可変（設定画面で大/特大切替）、コントラスト比 WCAG AA以上、タップ領域44px以上 |
| パフォーマンス | 初回表示3秒以内（4G相当）、Service Workerによるオフライン起動対応 |
| 操作性 | 主要操作（チェック実行、項目追加、予定登録）は3タップ以内 |
| 互換性 | iOS Safari／Android Chrome の最新2バージョンを正式サポート |
| プライバシー | 個人データは端末内のみに保存し、外部送信しない |
| 多言語対応 | MVPは日本語のみ。文言は将来のi18n化を見据えてハードコードせず一箇所に集約する |

---

## 8. リスクと前提

- 通知の信頼性はブラウザ・OS依存であり100%保証はできない（5章）。
- localStorageの容量制限（端末・ブラウザ依存、目安5MB前後）はMVP規模のデータ量では問題にならない想定。
- GitHub Pagesでのカスタムドメイン・HTTPS配信は標準対応のためPWAインストール要件を満たす。
