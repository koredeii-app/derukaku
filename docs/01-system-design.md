# システム設計書 — デルカク✓

## 1. 概要

| 項目 | 内容 |
|---|---|
| アプリ名 | デルカク✓（出るまえ確認） |
| サブタイトル | 出発前の忘れ物チェック |
| 目的 | 出発前の最終確認を簡単・確実に行い、忘れ物を防止する |
| 形態 | PWA（単体アプリ、共通エンジンは使用しない） |
| バックエンド | なし（サーバーレス、全データはローカル保存） |
| 主要ターゲット | 忘れ物が多い人／学生／社会人／子育て世代／高齢者／ADHD傾向のある人／通院・介護で持ち物管理が必要な人 |

非機能要件として「シンプル・迷わない・大きな文字・大きなボタン・高齢者でも使える・3タップ以内で主要操作完了」を最優先する。

## 3. 技術スタック

| レイヤー | 技術 | 採用理由 |
|---|---|---|
| UI | React 18 | コンポーネント単位での開発、将来のCapacitor移行との親和性 |

| 永続化 | localStorage（`zustand/middleware` の `persist`） | 初期リリースの要件に合致。将来IndexedDBへ移行しやすい抽象化を設ける |

## 5. 通知機能の技術方針（重要な制約事項・2026-06-19更新）

ブラウザ標準のPWAには、サーバーなしで「アプリを閉じていても指定時刻に確実に通知する」ことに技術的な制約がある。実装・検証の結果、以下が判明した。

- アプリ（タブ）を開いている間は、`setTimeout` ベースのタイマーで指定時刻に正確に通知処理を発火できる（実機検証済み）。
- アプリを完全に閉じている場合、JSタイマーは消滅するため通知は届かない。
- この問題を「サーバーなし」のまま解決する標準APIとして存在した `Notification Triggers API` はChromeが試験提供後に廃止し、現在は使用不可。`Periodic Background Sync` は対応ブラウザが限られ、発火間隔も粗く（最短でも数時間単位）、指定時刻通知には使えない。
- `Web Push` で閉じた状態でも届けることは可能だが、「指定時刻に送信する」ためのスケジューラとしてサーバーが必須になり、本プロジェクトの「サーバーを使用しない」方針と矛盾する。

これらを踏まえ、**「アプリを閉じていても確実に通知する」要件は、Capacitorによるネイティブアプリ化を前倒しして対応する**方針とした（詳細は5.1）。PWA単体での確実な閉鎖時通知は実現不可能という前提に立つ。

### 5.1 段階的な実装方針
1. **MVP（PWA, 現フェーズ）**: `Notification` APIによるアプリ起動中の通知。閉じている間は届かない制約を許容し、ホーム画面での「本日の未完了チェック」表示で補完する。
2. **Capacitorネイティブ化（前倒し実施）**: `@capacitor/local-notifications` を用いたOS標準のローカル通知に切り替える。サーバー不要のまま、アプリが完全に閉じていても・端末を再起動しても指定時刻に確実に発火する。Android/iOSへのネイティブパッケージングが前提となるため、ストア配信（Google Play / App Store）が必要になる。
3. 通知発火ロジックは `src/lib/notification.ts` に `Capacitor.isNativePlatform()` による分岐を持たせ、ネイティブ環境では `LocalNotifications`、Web/PWA環境では既存のWeb Notification実装にフォールバックする（同一コードベースを維持）。

### 5.2 データ構造との関係
データ構造（4章参照）の `ScheduleEntry.notification` は発火方法（Web Notification / ネイティブLocalNotifications）に依存しない汎用モデルとして設計済みのため、上記の切り替えに伴うデータ構造の変更は不要。

## 6. データ永続化方針

- 初期リリース: `localStorage` に、ストアごとにJSONを保存（キー設計は[04-data-structure.md](04-data-structure.md)参照）。
- スキーマには `schemaVersion` を持たせ、将来の構造変更時にマイグレーション関数で吸収する。
- 永続化処理は `src/lib/storage.ts` に抽象化し、将来 `IndexedDB` や `Capacitor Preferences/SQLite` に切り替える際もストア層のコードは変更不要にする。

## 7. 非機能要件

| 区分 | 要件 |
|---|---|
| アクセシビリティ | 文字サイズ可変（設定画面で大/特大切替）、コントラスト比 WCAG AA以上、タップ領域44px以上 |
| パフォーマンス | 初回表示3秒以内（4G相当）、Service Workerによるオフライン起動対応 |
| 操作性 | 主要操作（チェック実行、項目追加、予定登録）は3タップ以内 |
| 互換性 | iOS Safari／Android Chrome の最新2バージョンを正式サポート |
| プライバシー | 個人データは端末内のみに保存し、外部送信しない |
| 多言語対応 | MVPは日本語のみ。文言は将来のi18n化を見据えてハードコードせず一箇所に集約する |

---

## 8. リスクと前提

- 通知の信頼性はブラウザ・OS依存であり100%保証はできない（5章）。
- localStorageの容量制限（端末・ブラウザ依存、目安5MB前後）はMVP規模のデータ量では問題にならない想定。
- GitHub Pagesでのカスタムドメイン・HTTPS配信は標準対応のためPWAインストール要件を満たす。
