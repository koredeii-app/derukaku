# 改善提案 — デルカク✓

プロジェクトマネージャー／開発／マーケティングの3視点から、元仕様にない補足提案をまとめる。

## 1. 開発視点

### 1.1 通知の信頼性は「正直に伝える」設計にする（2026-06-19: 対応方針確定）
サーバーを持たないPWAでは、アプリを閉じた状態での通知が100%保証されない。実装・実機検証の結果、これはPWA単体では解決不可能な制約と確定したため、Capacitorによるネイティブ化を前倒しすることで対応する（[01-system-design.md 5章](01-system-design.md#5-通知機能の技術方針重要な制約事項2026-06-19更新)、[05-mvp-tasks.md 11章](05-mvp-tasks.md#11-capacitor%E3%83%8D%E3%82%A4%E3%83%86%E3%82%A3%E3%83%96%E5%8C%96%E5%89%8D%E5%80%92%E3%81%97%E5%AF%BE%E5%BF%9C2026-06-19%E8%BF%BD%E5%8A%A0)）。ネイティブ化完了までの暫定対応として、ホーム画面に「本日の未完了チェック」を表示するフォールバックは維持する。

### 1.2 データのバックアップ・引っ越し手段を早期に用意する
ローカル保存のみのため、端末の機種変更やブラウザのキャッシュ削除でデータが消える。MVPの段階で「設定画面からJSONを書き出し／読み込みできる」機能（[05-mvp-tasks.md](05-mvp-tasks.md) T91付近に追加候補）を入れておくと、将来のクラウド同期構想への橋渡しにもなる。

### 1.3 履歴データを最初から残す設計にする
「将来機能候補：AI提案（傘をよく忘れています、等）」を見据え、`CheckSession`（[04-data-structure.md](04-data-structure.md)）は最初から保存しておく。MVPでは履歴を見せるUIを作らなくても、データだけ蓄積しておけば後から機能追加するコストが大幅に下がる。

### 1.4 セット・項目の削除時の整合性
項目を削除した際、その項目を含むセットや予定が残っていると不整合が起きる。削除前に「このセットで使われています」と警告するか、参照を自動的に取り除くかを決めておく必要がある（T22で対応想定）。

## 2. UI/UX視点

### 2.1 オンボーディングを3タップ以内の例外にしない
「3タップ以内で主要操作完了」という方針に対し、初回のオンボーディング＋通知許可だけは例外として許容する設計が必要（許可ダイアログはOS側のUIのため制御不能）。ただしオンボーディング自体は最大3画面までに絞る。

### 2.2 チェック実行画面の「未チェック項目」を視覚的に強調
高齢者・ADHD傾向のあるユーザー向けに、チェック済み項目はグレーアウト、未チェック項目のみ明るい色で残すなど、残タスクが一目でわかる配色にする。

## 3. マーケティング視点

### 3.1 ASO（ストア最適化）・SEOキーワード案
「忘れ物防止」「持ち物チェック」「出発前チェックリスト」「忘れ物 アプリ」「ADHD 持ち物管理」などのキーワードを説明文・LPに織り込む。「デルカク」という語感はそのままでは検索されにくいため、説明文の冒頭に「出発前の忘れ物防止アプリ」を必ず入れる。

### 3.2 差別化訴求文言（ドラフト）
> 「ただのチェックリストじゃない。出発前の“最後の1分”に特化した、忘れ物防止アプリ。」

タスク管理アプリ／TODOアプリとの違いを訴求の起点に置く（仕様書「差別化ポイント」を訴求文に直接反映）。

### 3.3 ターゲット別の入口を分ける
高齢者向け・ADHD傾向のある人向け・子育て世代向けでは刺さる言葉が異なる。SNS施策では同一アプリでも訴求軸を分けた複数パターンの投稿案を用意するのが望ましい（例: 高齢者向けは「ご家族の安心」、ADHD向けは「忘れ物ゼロを目指す仕組み」、子育て世代向けは「子どもの持ち物も一緒に管理」）。

### 3.4 プライバシー訴求も差別化要素になる
「データは端末内のみ・サーバーに送信しない」という設計上の特徴は、プライバシー意識の高い層への訴求材料として説明文・LPに明記する価値がある。

## 4. 法務・信頼性視点

### 4.1 プライバシーポリシーページの設置
個人情報を外部送信しない設計だが、ストア公開やPWA公開時にはプライバシーポリシーページの有無が信頼性・審査通過に影響するため、簡潔な1ページを用意する（[05-mvp-tasks.md](05-mvp-tasks.md) T103）。
